# 项目启动流程（Project Kickoff）

把一个"老板拍脑袋/灵光一现"的模糊想法，变成一份**想清楚、写清楚、对齐清楚**的可执行项目。这套流程不是凭空发明的，而是把 Amazon、Google、Meta、Anthropic/OpenAI 和传统企业（PMBOK / Stage-Gate）启动大项目时反复验证有效的做法蒸馏出来。

**这个 skill 的产出不是"开个会就完事"，而是一套启动包（kickoff packet）：一份能拍板的启动文档 + 决策权清单 + 里程碑关卡 + 启动会议程。**

详细模板见 `references/templates.md`（直接抄填）；各大厂具体怎么做、为什么这么做见 `references/playbooks.md`（用户想深入了解时再读）。

## 铁律：先问清楚，再给方案（硬门）

**IMPORTANT：在给出任何启动方案、文档、架构或计划之前，必须先反问用户、收集到足够的关键信息、资料和数据。信息没拿到，不准直接出方案。**

这是本 skill 的第一道硬门，优先级高于一切后续步骤：

- 用户一句话抛来一个想法时，**默认动作是"先问"，不是"先答"**。先按第 1 步的必问清单做结构化访谈。
- 一次只聚焦少量关键问题，优先给选择题/示例，不要一口气甩 20 个问题把人问懵。
- **缺失的信息绝不替用户脑补后写进方案**。拿不到就在文档里显式标 `TBD（负责人 / 截止日）`，并告诉用户"这块缺什么、为什么重要、不补会有什么风险"。
- 只有在以下情况才可跳过提问直接给方案：① 用户已经提供了足够输入（必问清单基本齐全）；② 这是 S 小型可逆任务，问与不问代价都很低；③ 用户明确说"先别问，直接按假设给草案"——此时也要把所采用的假设单列出来。
- 需要用户提供的"资料 / 数据"包括但不限于：现有文档、竞品/对标信息、真实业务数据、公司主体与合规材料、已有代码或系统、预算与时间约束、干系人名单。**没有这些，方案就是空中楼阁，要主动开口要。**

> 一句话：这个 skill 不是"接到需求就输出文档"，而是"先把需求问透、把资料要齐，再输出文档"。

## 核心理念：大厂的 5 条共识

不管是 AI 大厂还是传统软件公司，靠谱的项目启动都绕不开这五条。开工前先用它们自检：

1. **从结果倒推，不从方案出发（Working Backwards）。** 先回答"成功长什么样、客户拿到手什么感受"，再回头设计怎么做。Amazon 甚至先把上线那天的"新闻稿"写出来。先有方案后补目标，是失败项目的头号特征。

2. **用文字想清楚，别用 PPT 糊弄（Narrative over slides）。** Amazon 用六页纸叙事代替 PPT，Google/Meta 用 design doc，Uber/Stripe 用 RFC。原因一致：**完整的句子会逼你暴露思维漏洞，bullet point 会帮你藏起来。** 写得出来才算想明白。

3. **启动前就定义"可衡量的成功"。** 没有量化的成功指标和验收标准，项目就没有"做完"的定义，只会无限拖延和扯皮。这一步在传统流程叫验收标准，在大厂叫 success metrics / OKR，在 AI 项目叫 eval 指标。

4. **明确决策权与责任（DACI / RACI）。** 大项目翻车常常不是因为没人干活，而是因为**没人拍板**或者"人人都觉得别人会管"。启动时就要白纸黑字写清楚：谁拍板（D/A）、谁干活（R）、谁要被咨询（C）、谁只需知会（I）。

5. **分阶段、设关卡，不要一次性 all-in（Phase Gate）。** 传统企业叫 Stage-Gate，AI 大厂叫 gated rollout / 灰度放量。本质相同：把大赌注拆成几段，每段结束设一道关卡，达标才放行下一段，不达标就砍掉或调整。**启动 ≠ 承诺做完，启动 = 承诺做完第一阶段并在关卡处重新评估。**

## 操作流程（5 步）

按顺序走。每一步先判断该做到什么深度，不要对小项目上重型流程，也不要对大项目省关键步骤。

### 第 0 步：判断项目体量，选启动深度

先别急着填模板。用 T-shirt sizing 给项目定个量级，决定流程轻重——**对大厂方法最大的误用，就是给一个两周的小活套一套企业级立项流程。**

| 量级 | 典型特征 | 最少要产出 |
|------|---------|-----------|
| **S 小型** | 1 人、≤2 周、可逆、不跨团队 | 一段话写清：目标 + 验收标准 + 负责人。其余跳过。 |
| **M 中型** | 2–6 人、1–2 个月、跨 1–2 个团队 | 一页项目章程 + RACI + 里程碑 + 一次启动会 |
| **L 大型** | 多团队、季度级以上、不可逆/高投入/对外发布 | PR/FAQ 或六页纸 + 设计文档/RFC + RACI + 阶段关卡 + 风险登记 + 正式启动会 |

判断不准时，宁可问用户一句："这个项目大概几个人、做多久、出了问题好不好回头？"再定深度。

### 第 1 步：收集启动输入（结构化访谈）—— 出方案前的硬门

启动文档质量取决于输入质量。**这一步是"铁律：先问清楚，再给方案"的落地：没走完这步、关键信息没拿到，不要进入第 2 步出方案。** 用下面这组高杠杆问题访谈用户（或干系人）。**关键纪律：遇到含糊回答不要替用户脑补后写进文档——把含糊点标出来，这正是启动阶段最该暴露的风险。**

提问方式：一次聚焦少数几个最关键的问题，优先给选择题或示例答案，根据回答再追问；不要一次性把整张清单甩给用户。除必问清单外，**主动向用户索取相关资料和数据**：现有文档、竞品/对标、真实业务数据、公司主体与合规材料、已有代码或系统、预算与时间表、干系人名单。

必问清单：
- **问题 / 机会**：要解决谁的什么痛点？不做会怎样？
- **为什么是现在（Why now）**：什么变化让这件事此刻值得做？
- **目标客户 / 受益人**：具体是谁，不是"所有人"。
- **成功指标**：怎么量化"成了"？给出可测的数字或验收标准。
- **范围（In / Out）**：明确写"这次做什么"和"这次不做什么"。**写清不做什么，比写做什么更能防止失控。**
- **干系人与决策人**：谁拍板、谁出钱、谁要被知会、谁会反对。
- **依赖与约束**：依赖哪些团队/系统/外部条件？预算、deadline、合规红线。
- **主要风险与假设**：最可能让项目翻车的三件事是什么？哪些是还没验证的假设？

输入不全也能往下走，但要在文档里把空缺显式标成"待确认（TBD）+ 负责人 + 截止日"，不要假装齐了。

### 第 2 步：产出启动文档

按第 0 步的量级，选对应产物，从 `references/templates.md` 取模板填写：

- **M 中型** → 一页项目章程（One-page Project Charter）。最通用，传统软件公司和大厂内部小项目都用。
- **L 大型 / 对外产品** → **PR/FAQ（Working Backwards）**：先写一篇"上线那天的新闻稿"+ 客户和内部各一组 FAQ。强迫从客户价值倒推。
- **L 大型 / 需要高管拍板** → **六页纸叙事（6-pager）**：用完整段落而非 bullet 讲清背景、目标、原则、方案、风险。
- **偏技术 / 架构选型** → **设计文档 / RFC**：方案、备选方案与取舍、影响面、回滚预案，开放给同行评审。
- **AI / 模型 / 数据类项目**：在以上基础上补三样——**评估方案（eval / 验收数据集）、分阶段放量计划（灰度 → 全量）、安全与失败模式分析**。这是 AI 大厂区别于传统软件的关键，别漏。

每份文档都必须包含：目标、可衡量的成功指标、范围 In/Out、决策权、里程碑、风险。其余按需。

#### 2.x 落地项目文件（AGENTS.md 及配套）

启动文档是"给人对齐"，还要顺手把"给 AI 代理和后续开发读"的项目文件建起来，让方案能直接落到仓库里。**默认产出 `AGENTS.md`，并按需补齐配套文件：**

- **`AGENTS.md`（首选，业界中立标准）**：放仓库根目录，给任何 AI 编码代理读的项目说明——项目目标、目录结构、启动/测试/构建命令、代码规范、修改边界、验收标准。模板见 `references/templates.md` 第 9 节。
- **`CLAUDE.md`**：如果用户主要用 Claude Code，可直接写 `CLAUDE.md`，或写一行 `参见 AGENTS.md` 指过去，避免两份重复维护。
- **`README.md` 骨架**：项目简介、快速上手、目录说明。
- **`docs/` 规范目录骨架**：按需创建 `docs/api.md`、`docs/db.md`、`docs/frontend.md`、`docs/release.md` 等占位，复杂规范不要塞进 AGENTS.md。
- **`.claude/launch.json`**（可运行项目）：预览/启动配置。

裁剪原则：S 小型项目通常只需一个 `AGENTS.md`；M/L 项目再按需补 `README`、`docs/` 和 `CLAUDE.md`。生成前先确认目标目录，**不要覆盖用户已有的同名文件**，发现已存在先提示再决定合并或保留。

### 第 3 步：对齐评审（启动会）

文档是为了对齐，不是归档。开一次启动会把人对齐到同一页：

- **学 Amazon 的"先静读后讨论"**：会议前 10–20 分钟让所有人**当场默读**文档，再开始讨论。比起会前发出去（没人看），当场读能保证每个人真的在同一信息基线上。
- 议程见 `references/templates.md` 的"启动会议程"。核心环节：背景与目标 → 范围与不做什么 → 角色与决策权（过一遍 RACI）→ 里程碑与关卡 → 风险与依赖 → 当场敲定 owner 和 next step。
- **会议产出必须落字**：决策、负责人、待办、截止日，会后发给所有人（含缺席者）。口头共识不算数。

简单内部项目 30–45 分钟，复杂多干系人项目 60–90 分钟。

### 第 4 步：设定阶段关卡（Gates）

启动不是开闸放水。给项目划出 2–4 个阶段，每个阶段末设一道关卡：

- 每道关卡写清**放行标准（exit criteria）**：达到什么指标/产出才进入下一阶段。
- 在关卡处只有三种决定：**继续（Go）/ 调整再来（Recycle）/ 砍掉（Kill）**。能砍掉是这套机制的价值所在——及早止损好过沉没成本越陷越深。
- AI 项目的关卡通常是 eval 指标达标、灰度放量无重大问题、安全审查通过。
- 把关卡和里程碑写进启动文档，启动会上同步。

## 完成标准：启动包清单

一次合格的项目启动，交付物应覆盖（按量级裁剪）：

- [ ] 出方案前已完成结构化访谈、关键信息/资料/数据已收齐或显式标 TBD（硬门，见上文铁律）
- [ ] 启动文档（章程 / PR-FAQ / 六页纸 / 设计文档，至少一份）
- [ ] 可衡量的成功指标 / 验收标准
- [ ] 范围 In / Out 清单
- [ ] RACI 或 DACI 决策矩阵（谁拍板写明）
- [ ] 里程碑 + 阶段关卡（含放行标准）
- [ ] 风险登记表（Top 风险 + 应对 + owner）
- [ ] 启动会议程 + 会后纪要（决策/待办/owner/截止日）
- [ ] 项目文件已落地：`AGENTS.md`（必备），按需 `README` / `docs/` 骨架 / `CLAUDE.md` / `.claude/launch.json`
- [ ] 所有 TBD 都挂了 owner 和截止日

逐项确认，缺哪项就明确告诉用户缺什么、为什么重要，而不是默默跳过。

## 进阶：启动之后怎么落地（Superpowers 三件套）

启动包定的是"做什么、为什么、谁拍板"；真正动手前后还有三步，来自 obra/superpowers 这套被验证有效的方法论，正好接在启动之后。完整精华见 `references/superpowers.md`。

铁律是**按序串联，且设计没获批前不动手**：

1. **brainstorming（先想清楚）** —— 动手前先对话澄清意图。一次只问一个问题（优先选择题），给 2–3 个方案讲清取舍，把设计分段确认后写成 spec。硬门：设计没被批准，不写代码、不搭脚手架、不做任何实现。这是对"第 1 步收集输入"的升级打法。
2. **writing-plans（写成可执行计划）** —— 假设执行人对项目零了解，把成功所需的一切写进计划：任务切到 2–5 分钟一个、每个任务有独立可验收的交付物、**禁占位符**（不准 TBD、"处理一下边界"这种空话）。口诀 DRY、YAGNI、TDD、频繁提交。
3. **subagent-driven-development（拆任务交付 + 双重验收）** —— 每个任务派一个上下文干净的执行体，交付后必须过两道验收：①规范符合（该做的做了、没多做）②质量过关；有问题就修了再评，最后用最强模型做一次整体评审。

非代码项目同样适用：把"代码 / subagent"换成"可执行内容 / 具体执行人"，精华不变——**想透再动手、计划不留空话、交付要双重验收**。

## 想深入了解大厂具体怎么做

`references/playbooks.md` 里有一张"公司 × 做法 × 适合谁"对照表，以及 Amazon Working Backwards/六页纸、Google/Meta design doc + OKR、Uber/Stripe/Squarespace 的 RFC 文化、Anthropic/OpenAI 的门控发布、传统企业 Stage-Gate/PMBOK 立项的逐一拆解。用户想比较"AI 大厂和传统软件公司启动流程差在哪"时，读这个文件再回答。

**一句话差异**：传统软件公司启动重在**控风险**（先评审、签字立项、按关卡放行）；AI 大厂启动重在**降不确定性**（先写清客户价值、用文字逼自己想透、用 eval 和灰度放量边做边验证）。好的启动流程两者都要：既写清楚要去哪，又留好随时止损的关卡。
