# Superpowers 三件套：从想法到落地

源自 GitHub 上的 **obra/superpowers**（作者 Jesse Vincent）。三个 skill 串成一条完整链路，正好接在项目启动之后：**启动包定了"做什么 / 为什么 / 谁拍板"，这三件套管"怎么把它想透、写成可执行计划、并高质量执行"。**

## 串联铁律（必须按序，且设计没获批不准动手）

```
brainstorming（产出 design spec）
   → writing-plans（产出可执行 plan）
      → subagent-driven-development（拆任务执行 + 双重验收）
```

brainstorming 之后唯一能进入的就是 writing-plans。**跳过设计直接写代码 / 搭脚手架 / 动手，是这套方法明令禁止的头号反模式。**

---

## 1. brainstorming —— 先想清楚再动手

**核心理念**：Design before implementation。任何项目（无论看起来多简单）动手前都要先经过协作式对话，澄清意图、需求与设计，避免被未经检验的假设带偏、白做。

**精华做法**：
- 先探查上下文（相关文件、文档、历史记录），别一上来就问。
- **一次只问一个问题，优先给选择题**，不要一口气抛一堆问题砸晕用户。
- 给出 **2–3 个方案 + 各自取舍**，让用户选，而不是直接给一个答案。
- 把设计**分段呈现、逐段确认**，再写成 design spec 落到文件。
- 自检 spec：有没有占位符、自相矛盾、含糊、范围蔓延（scope creep）。
- **硬门（hard gate）**：设计没被批准前，不写代码、不搭脚手架、不做任何实现动作。
- 多子系统的需求，先拆成独立子项目，再逐个细化。

**迁移到非代码项目**：把"design spec"换成"启动文档 / 方案"，规则完全通用——想清楚、写下来、获批准，再动手。这本质上是本 skill"第 1 步收集输入"的升级打法。

---

## 2. writing-plans —— 把方案写成"零上下文也能执行"的计划

**核心理念**：假设执行人对你的项目和代码库**一无所知**，然后把他成功所需的一切都写进计划。口诀：**DRY（不重复）、YAGNI（不做用不上的）、TDD（测试驱动）、频繁提交**。

**精华做法**：
- **范围检查**：一个计划只覆盖一个内聚的子系统；多子系统要拆成多个独立计划。
- 写任务前**先规划文件结构**：每个文件单一职责，一起改的东西放一起。
- **任务右尺寸（right-sizing）**：一个任务 = 值得一个新评审员单独把关、且能**独立验收**的交付物。该被独立否决/接受的，才拆成两个任务。
- **步骤切到 2–5 分钟一个离散动作**：写测试 → 跑测试看它失败 → 写最小实现 → 跑测试看通过 → 提交。
- **禁占位符（No Placeholders）**：不准写 TBD、"处理一下边界情况"、"写点测试"这类空话；**凡是改代码的步骤，必须给出完整代码**。
- 自检三连：① 每条需求都能指到某个任务（查覆盖、标缺口）② 扫一遍占位符红旗 ③ 跨任务的命名/签名/类型是否一致。
- 存到固定路径并交接，让执行方（subagent 或本人）接手。

**迁移到非代码项目**：把"完整代码"换成"完整可执行内容"，原则不变——颗粒度小、可独立验收、不留空话。

## 3. subagent-driven-development —— 拆给"干净上下文"的执行体 + 双重验收

**核心理念**：每个任务派一个**全新的 subagent**（上下文隔离、不被前面的对话污染）+ 任务级双重评审 + 最后整体评审 = 又快又稳。避免上下文污染，且任务之间无需人工交接、可连续推进。

**精华做法**：
- **预检（pre-flight）**：第一个任务开工前，通读整份计划一遍，先把矛盾挑出来。
- **每个任务一个循环**：派实现者 subagent（给足场景上下文 + 任务简报 + 上游接口 + 验收契约）→ 先答疑再开工 → 实现 + 测试 + 自检 + 提交 → 生成评审包，派评审者。
- **两阶段评审（缺一不可）**：① **规范符合**（该做的做了、没有多做）② **代码质量**（测试覆盖、清晰、可维护）。
- 评审发现 Critical / Important 问题 → 派修复 subagent → 再评审；**规范 ✅ 且质量过关，才在进度账本里标完成。**
- **别忽略升级求助，别让同一个模型不做任何改动就重试。** 派任务时显式指定模型，产物用文件传递而非粘贴大段文本。
- 全部任务完成后，**用最强模型对整支分支做一次整体评审**，再收尾。

**迁移到非代码项目**：把"subagent"换成"把活拆给具体执行人 / 外包 / 同事"，精华是——给执行人一个**干净完整的任务包**，交付后做两道验收：**"对不对（符合要求）"和"好不好（质量）"**。

## 来源与适用边界

- 来源：GitHub `obra/superpowers`，skills 目录下的 `brainstorming` / `writing-plans` / `subagent-driven-development`。自 v5 起也支持 Codex、Cursor、Gemini CLI 等。
- **适用边界**：原版为**软件开发**场景设计（含 TDD、git commit、代码评审等强约束）。用在非代码项目时，取其**方法论精华**即可——想透再动手、计划不留空话、交付要双重验收——不必照搬代码相关的具体步骤。
