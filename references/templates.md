# 启动文档模板合集（直接抄填）

按项目量级取用。每个模板都标了"什么时候用"。填写时把示意文字替换成真实内容；填不出来的地方标 `TBD（负责人 / 截止日）`，不要硬编。

## 目录
1. 一页项目章程（One-page Project Charter）— M 中型首选
2. PR/FAQ（Working Backwards）— L 大型 / 对外产品
3. 六页纸叙事（6-pager）— 需要高管拍板
4. 设计文档 / RFC — 技术方案与选型
5. RACI / DACI 决策矩阵 — 所有 M/L 项目
6. 启动会议程（Kickoff Agenda）
7. 风险登记表（Risk Register）
8. 里程碑与阶段关卡（Milestones & Gates）
9. AGENTS.md（给 AI 代理读的项目说明）+ 配套文件

---

## 1. 一页项目章程（One-page Project Charter）

**何时用**：中型项目、传统软件公司立项、大厂内部小项目。最通用，控制在一页。

```
项目名称：
负责人（owner）：            发起人 / 出资人（sponsor）：
日期：                       目标量级：S / M / L

一、为什么做（背景与问题）
- 要解决谁的什么问题？不做会怎样？为什么是现在？

二、目标与成功指标（可衡量）
- 目标：
- 成功指标：______（必须是可测的数字或明确验收标准，如"X 降到 Y""上线后 30 天 Z 达到 N"）

三、范围
- 本次做（In）：
- 本次不做（Out）：        ← 必填，防失控

四、关键干系人与决策权
- 拍板人：        出资人：        主要执行：        需知会：

五、里程碑与关卡
- M1（日期）→ 关卡放行标准：
- M2（日期）→ 关卡放行标准：

六、主要风险与依赖
- 风险1 / 应对 / owner：
- 关键依赖：

七、预算 / 资源 / 约束
- 人力：        预算：        deadline / 合规红线：

八、待确认（TBD）
- 事项 / owner / 截止日：
```

## 2. PR/FAQ — Working Backwards（Amazon）

**何时用**：大型项目、对外产品/功能、想从客户价值倒推。**先写新闻稿，再写 FAQ。** 写不出让人兴奋的新闻稿，往往说明这事不值得做。

### Part A：新闻稿（Press Release，约 1 页，用"已上线"的口吻写）

```
标题：[产品名]，一句话让目标客户秒懂这是什么
副标题：[谁] 现在可以 [获得什么核心收益]

[首段·摘要] 一段话讲清：产品是什么、给谁、最大价值。像已经上线那样写。

[问题段] 今天客户面临什么问题，现状有多糟。

[方案段] 我们的产品如何解决，凭什么比现有方案"更快/更省/更好"。

[客户体验段] 客户怎么用、用了之后生活/工作有什么不同。

[引语] 一句客户证言（虚拟）+ 一句公司负责人的话。

[行动召唤] 客户从哪开始用。
```

判据：如果新闻稿描述的东西并不比现状**明显更快、更省、更好**，或没带来体验上的台阶式提升，就别做。

### Part B：FAQ（分两组）

```
客户 FAQ（客户会问的）：
- 多少钱 / 怎么获取 / 和 X 有什么区别 / 数据安全吗 / 不好用怎么办 ...

内部 FAQ（团队和高管会问的，越尖锐越好）：
- 客户为什么需要它？市场多大？
- 最难的技术/运营挑战是什么？要多少人多少钱多久？
- 怎么衡量成功（指标）？多久能见效？
- 最大的风险和未验证假设是什么？为什么我们能赢？
- 为什么是现在做、为什么是我们做？
```

提示：Amazon 团队常把 PR/FAQ 改十稿以上、和高管碰五次以上才定稿。第一版不完美是正常的。

## 3. 六页纸叙事（6-pager / Narrative Memo，Amazon）

**何时用**：需要高管或评审委员会拍板的大决策。**用完整段落写，禁用 bullet 糊弄。** 会上先让大家静读 20 分钟再讨论。

六个部分：

```
1. 引言（Introduction）—— 一两段把"这份文件要决定什么"讲清楚。
2. 目标（Goals）—— 本项目要达成什么，附关键指标（收入/成本/满意度/采用率…）。
3. 原则（Tenets）—— 指导取舍的几条基本信念，遇到冲突时按它们决策。
4. 现状（State of the Business / Background）—— 相关背景、数据、当前处境。
5. 经验教训（Lessons Learned）—— 过去类似尝试学到了什么，避免重蹈。
6. 战略重点（Strategic Priorities）—— 全文重心：具体怎么做，每个动作如何指向第 2 部分的目标。
```

写作要求：能让一个没参与过的高管，静读一遍就能听懂全貌并提出尖锐问题。把读者会问的反对意见提前在文中回应掉。

## 4. 设计文档 / RFC（Google design doc / Uber-Stripe RFC）

**何时用**：技术方案、架构选型、跨团队接口变更。核心价值是**把决策写下来 + 开放同行评审**，让隐藏的设计缺陷在动工前被人挑出来。

```
标题 / 作者 / 状态（草稿·评审中·已批准·已废弃）/ 评审人

1. 背景与问题（Context）—— 为什么要做，约束是什么。
2. 目标 / 非目标（Goals / Non-Goals）—— 明确不解决什么。
3. 提议方案（Proposed Design）—— 架构、数据流、接口、关键决策。
4. 备选方案与取舍（Alternatives Considered）—— 还考虑过哪些，为什么不选。 ← 评审最看重这节
5. 影响面（Impact）—— 对现有系统/团队/性能/成本/安全的影响。
6. 风险与回滚（Risks & Rollback）—— 出问题怎么退回去。
7. 测试与上线计划（Test & Rollout）—— 怎么验证、怎么灰度。
8. 开放问题（Open Questions）—— 还没想清楚、需要评审者帮忙的点。
```

流程提示：作者写草稿 → 指定评审人 + 公开给相关团队 → 评审用"Yes, if（可以，前提是…）"而非"No, because"推动方案改进 → 达成一致后标记 Approved 并存档可检索。

## 5. RACI / DACI 决策矩阵

**何时用**：所有 M/L 项目。防止"没人拍板"和"人人以为别人会管"。

RACI：每项关键工作/决策，标清四种角色——
- **R（Responsible）** 实际干活的人
- **A（Accountable）** 最终负责、拍板的人（每项**只能有一个 A**）
- **C（Consulted）** 要事先咨询的人
- **I（Informed）** 只需事后知会的人

```
| 工作 / 决策     | 张三 | 李四 | 王五 | 产品总监 |
|----------------|------|------|------|---------|
| 技术方案定稿    | R    | C    |      | A       |
| 预算审批        |      |      | R    | A       |
| 上线放行        | C    | R    | C    | A       |
```

DACI（更偏单次决策）：**D**river 推动者 / **A**pprover 拍板者 / **C**ontributors 贡献者 / **I**nformed 知会者。一个决策一行写清这四角色即可。

## 6. 启动会议程（Kickoff Agenda）

**何时用**：M/L 项目正式开工会。简单内部会 30–45 分钟，复杂多干系人会 60–90 分钟。

```
0. （可选·学 Amazon）当场静读启动文档        10–20 分钟
1. 背景与目标：为什么做、成功长什么样          5–10 分钟
2. 范围：做什么 / 不做什么                     5 分钟
3. 角色与决策权：过一遍 RACI，确认每个 A       5–10 分钟
4. 计划：里程碑、阶段关卡、关键日期            10 分钟
5. 风险与依赖：Top 风险、卡点、需要的资源       10 分钟
6. 沟通机制：例会节奏、文档在哪、怎么同步进度    5 分钟
7. 当场敲定：行动项 + owner + 截止日           5–10 分钟

会后 24 小时内：发出纪要（决策 / 待办 / owner / 截止日），含缺席者。
```

参会人：核心执行成员 + 发起人/出资人 + 有审批权的决策人 + 关键跨团队伙伴。

## 7. 风险登记表（Risk Register）

**何时用**：M/L 项目。启动时先列 Top 5–10，之后定期更新。

```
| # | 风险描述 | 可能性(高/中/低) | 影响(高/中/低) | 应对策略 | owner | 状态 |
|---|---------|----------------|---------------|---------|-------|------|
| 1 |         |                |               | 规避/减轻/转移/接受 |  |  |
```

提示：把"还没验证的关键假设"也当风险登记——假设被证伪，往往是项目翻车的真正原因。

## 8. 里程碑与阶段关卡（Milestones & Gates）

**何时用**：M/L 项目，尤其不可逆/高投入/AI 灰度类。

```
阶段 1：[名称]（日期区间）
- 交付物：
- 关卡放行标准（exit criteria）：达到 ___ 才进入阶段 2
- 关卡决定：Go（继续）/ Recycle（调整再来）/ Kill（砍掉）

阶段 2：[名称]（日期区间）
- 交付物：
- 关卡放行标准：

…（AI 项目典型关卡：eval 指标达标 → 内测灰度无重大问题 → 安全审查通过 → 全量放开）
```

关卡的意义是**给"及时止损"留出制度化的决策点**。能在关卡处砍掉一个不该继续的项目，和能放行一个好项目同样重要。

## 9. AGENTS.md（给 AI 代理读的项目说明）+ 配套文件

**何时用**：启动落地阶段，把方案变成仓库里可执行的文件。`AGENTS.md` 是业界中立标准（多家 AI 编码工具通用），放仓库根目录，让任何 AI 代理一进项目就知道怎么干、边界在哪。**先填好启动文档再来生成它——内容大量复用启动文档里已经想清楚的目标、范围、验收标准。**

填写纪律：每一项都写成具体可执行的命令或规则，不要写"按需配置""见相关文档"这种空话；填不出来的标 `TBD（负责人 / 截止日）`，不要硬编。生成前确认目标目录，**不要覆盖已有同名文件**。

### AGENTS.md 模板

```md
# AGENTS.md

> 给 AI 编码代理的项目说明。人类读 README.md，代理读这份。

## 项目目标
- 这个项目用于：
- 主要用户和场景：
- 当前阶段（S/M/L、第几阶段）：
- 暂时不做（Out of scope）：

## 技术栈与目录结构
- 技术栈：
- 源码目录：
- 测试目录：
- 文档目录：docs/
- 禁止随意新增文件的位置：

## 开发命令
- 安装依赖：
- 本地启动：
- 运行测试：
- 代码检查 / lint：
- 构建：

## 代码规范
- 命名 / 风格：
- 提交信息规范：
- 不要引入不必要的依赖；优先复用现有库与工具。

## 修改边界（红线）
- 不要修改密钥、环境变量、生产配置。
- 不要删除用户数据、迁移文件、历史记录。
- 不要自动执行 git push 或部署。
- 破坏性 / 批量 / 不可逆操作前，先停止并向人确认。

## 验收标准（Definition of Done）
- 改完说明改了什么、在哪。
- 能跑测试必须跑，跑不了说明原因。
- 标出仍存在的风险。
- 验收指标（复用启动文档的成功指标）：

## 按需阅读的规范文档
- API 规范：docs/api.md
- 数据库规范：docs/db.md
- 前端规范：docs/frontend.md
- 发布流程：docs/release.md
```

### 配套文件（按项目量级裁剪）

- **CLAUDE.md**：用户主用 Claude Code 时。要么单独写，要么只放一行 `本项目代理规则见 AGENTS.md`，避免两份重复维护。
- **README.md 骨架**：

  ```md
  # 项目名

  一句话简介。

  ## 快速上手
  - 安装：
  - 启动：
  - 测试：

  ## 目录结构
  - src/ …
  - docs/ …

  ## 文档
  - 项目协作规则见 AGENTS.md
  ```

- **docs/ 骨架**：按需创建 `docs/api.md`、`docs/db.md`、`docs/frontend.md`、`docs/release.md` 占位，复杂规范从 AGENTS.md 拆出来放这里。
- **.claude/launch.json**（可运行项目的预览/启动配置）：

  ```json
  {
    "version": "0.0.1",
    "configurations": [
      { "name": "web", "runtimeExecutable": "npm", "runtimeArgs": ["run", "dev"], "port": 3000, "autoPort": true }
    ]
  }
  ```

裁剪原则：S 小型只需一个 AGENTS.md；M/L 再按需补 README、docs/、CLAUDE.md、launch.json。
