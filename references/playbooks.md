# 大厂启动方法详解（AI 大厂 × 传统软件公司）

用户想了解"各家具体怎么做""AI 大厂和传统软件公司差在哪"时读本文件再回答。下面先给一张对照表，再逐一拆解。

## 对照表：公司/流派 × 核心做法 × 适合谁

| 流派 / 公司 | 核心抓手 | 一句话精髓 | 最适合的场景 |
|------------|---------|-----------|-------------|
| **Amazon** | Working Backwards（PR/FAQ）+ 六页纸叙事 | 先写上线那天的新闻稿，从客户价值倒推 | 对外新产品、需要从 0 想清楚价值的大项目 |
| **Google / Meta** | Design Doc + OKR | 写设计文档让同行评审，用 OKR 对齐目标 | 工程项目、跨团队技术协作 |
| **Uber / Stripe / Squarespace** | RFC（Request for Comments） | 重要决策都写下来、公开评审、可检索 | 技术选型、架构变更、规模化团队 |
| **Anthropic / OpenAI（AI 大厂）** | eval 驱动 + 安全门控 + 分阶段放量 | 边做边用评估和灰度降低不确定性 | 模型/AI/数据类、结果不确定的项目 |
| **传统软件 / 企业（PMBOK）** | 项目章程 + 立项评审签字 | 先正式授权、定权责，再开工 | 合规要求高、跨部门、预算审批严的项目 |
| **传统制造 / NPD** | Stage-Gate（阶段-关卡） | 分 6 阶段，每段设关卡决定 Go/Kill | 高投入、长周期、不可逆的产品开发 |
| **敏捷团队** | Inception / Sprint Zero | 短平快对齐愿景、范围、风险后即开工 | 快速迭代、需求不稳定的软件项目 |

## 1. Amazon —— Working Backwards & 六页纸

**Working Backwards（逆向工作法）**：所有新产品先写一份 PR/FAQ（新闻稿 + 常见问答），从"客户读到这条新闻会不会兴奋到想买"出发，倒推该做什么。写新闻稿是一种**强制函数（forcing function）**：逼提出者把注意力锁死在客户价值上，而不是竞品、现有能力、损益表。限制篇幅本身也是强制函数——逼人成为更清晰的思考者和表达者。一个 idea 改十稿、和高管碰五次以上很常见。

**六页纸叙事**：2004 年 Bezos 一封邮件废掉了 PPT，改用六页叙事备忘录。开会时所有人先**静默读 20–30 分钟**再讨论。好处：决策更快更稳（每个人都吃透了信息）、冲突更少（详尽叙事提前回应了质疑）、项目起步更扎实（基础思考被充分表达）。结构：引言 / 目标 / 原则（tenets）/ 现状 / 经验教训 / 战略重点。

**为什么有效**：完整句子藏不住逻辑漏洞，bullet 可以。写作即思考。

---

## 2. Google / Meta —— Design Doc + OKR

**Design Doc**：动工前写设计文档，讲清背景、目标/非目标、方案、**备选方案与取舍**、影响面、测试与上线计划，开放给同行评审。重点不在文档本身，而在**评审过程**——让有经验的人在写代码前就挑出设计缺陷，成本最低。

**OKR（目标与关键结果）**：用 Objective（定性方向）+ Key Results（可量化结果）把跨团队目标对齐。启动项目时定 OKR，等于强制回答"成功怎么衡量"。

适合工程驱动、跨团队协作的项目。

---

## 3. Uber / Stripe / Squarespace —— RFC 文化

**RFC（Request for Comments）**：把重要技术决策写成文档，公开发出征求意见，达成共识后存档可检索。Uber 从几十人用到几千人工程团队，靠 RFC 打破信息孤岛、沉淀知识；规模过大（每周数百份）后又自建工具把审批人、审批任务、检索整合起来。Squarespace 用 RFC 做出过"把 Go 定为基础设施一等语言""切换到 gRPC"这类大决策，并提倡用 **"Yes, if（可以，前提是…）"** 代替 "No, because" 来推动方案改进而非一棍子打死。

**精髓**：Writing things down（把事情写下来）。口头决策无法规模化、无法检索、无法追责。

适合技术选型、架构变更、需要跨团队共识的场景。

## 4. Anthropic / OpenAI —— AI 大厂的启动特点

AI 大厂启动大项目（如前沿模型、Agent 产品）在传统软件流程之上，多了三件传统软件没有的事：

- **eval 驱动**：项目一开始就定好评估方案和验收数据集——"模型/系统在哪些指标上达到什么水平算成功"。eval 就是 AI 项目的成功指标和验收标准，没有 eval 的 AI 项目等于没有"做完"的定义。
- **安全门控（safety gating）/ gated rollout**：每次前沿能力发布都要有安全门控计划，先给少数组织灰度访问（如分阶段、受控放量），验证无重大问题再扩大。这是把风险关进笼子的制度。
- **分阶段放量（灰度 → 全量）**：不是一次性上线，而是内测 → 小范围灰度 → 逐步全量，每一步都是一道可叫停的关卡。

**和传统软件的根本差异**：传统软件的需求和结果相对确定，启动重在**控风险、按计划执行**；AI 项目结果高度不确定，启动重在**降低不确定性**——用 eval 量化"好不好"、用灰度边做边验证、为失败模式预留止损点。

## 5. 传统软件公司 / 企业 —— PMBOK 立项 & Stage-Gate

**PMBOK 启动过程组**：正式项目从**项目章程（Project Charter）**开始。章程定义目标、范围边界、成功标准、并正式授予项目经理权限。**没有签字的章程，项目就没有被正式授权。** 启动会通常在章程批准之后、动工之前开。会前 2–3 天把章程、范围说明、时间线、RACI、风险登记、沟通计划发给大家。

**Stage-Gate（阶段-关卡，又称 phase-gate / tollgate）**：把项目分成若干阶段（典型：构思 → 初步调研 → 商业论证 → 开发 → 测试 → 上市），每个阶段末有一道"关卡"评审，决定是否放行下一阶段。本质是**渐进式承诺（creeping commitment）**：不一次性投入全部资源，而是每过一关加一次码，随时可以 Kill。大企业、跨部门、高投入项目尤其适用。

**精髓**：先授权、定权责、控风险，分阶段放行。比起大厂的"先想清楚客户价值"，传统流程更强调"先建立正式的控制与问责机制"。

---

## 6. 敏捷 —— Inception / Sprint Zero

需求不稳定、要快速迭代的软件项目，启动常用轻量的 **Inception（启动工作坊）** 或 **Sprint Zero**：用 1–3 天快速对齐愿景、用户、范围、主要风险和第一周 backlog，不追求完整文档，够开工即可。适合 S/M 项目和创业团队。

---

## 怎么给用户选

- 想从 0 想清楚一个**对外产品/功能** → Amazon PR/FAQ。
- 需要**高管/委员会拍板**的大决策 → 六页纸叙事。
- **技术方案、架构选型、跨团队接口** → Design Doc / RFC。
- **AI / 模型 / 数据**类 → 在设计文档基础上加 eval 方案 + 灰度放量 + 安全分析。
- **合规严、跨部门、预算审批重**的企业项目 → PMBOK 项目章程 + 立项评审。
- **高投入、不可逆、长周期** → Stage-Gate 分阶段关卡。
- **小型、需求不稳、要快** → 敏捷 Inception，一页章程足矣。

实战中常常组合：比如一个 AI 产品大项目 = PR/FAQ（想清价值）+ 设计文档（技术方案）+ eval 与灰度计划（AI 特性）+ Stage-Gate 关卡（控投入）+ RACI（定权责）。
